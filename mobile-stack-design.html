Why 11 PRs

The mobile/responsive work grew as exploration — churn added and later undone, one concern spread across several PRs, two unrelated concerns sharing a commit. Epic #128 regroups the same final tree into one PR per concern: a touch shell that lands in one piece, a screens fix split into its shared-render half and its shell-specific half, build-info pulled out so its one real ordering constraint (the shell fetches build-info.json; without the generator the smoke gate fails) is explicit rather than implicit. Two independent tracks plus a disjoint tail — see “The 11 PRs” for the full map.

The shape before this work

Sidebar (20 cols), log (3 rows), and status (1 row) were fixed and unconditional; the map got whatever was left. There was no camera — the renderer drew world coordinates directly onto screen coordinates with zero offset, so anything past the map rect's edge was never drawn.

The concrete bug this fixes

xsofy's dungeon floors are 79×30 cells. The fixed layout's map area was 80×26 even at a common 30-row terminal — well under the 100×36 design cap. The bottom 4 rows of every floor were invisible and unreachable at any terminal shorter than 34 rows: nearly always, not on some edge case. Fixing this needed both a camera (to reach the rest of the world) and independent panel collapse (so the map has room to matter on a small screen).

The design, by feature

Camera & layout collapse

A camera (map-cell) offsets world coordinates so the visible window can move; it pans only once the player leaves a proportional dead zone — 60% of the map's width, 35% of its height — rather than tracking every step, and resets on a fresh run so restarts re-center. The three chrome panels collapse on independent thresholds instead of as one unit: sidebar hides below 80 game-columns, status below 67 columns or 2 rows, log below 14 rows. A 40×24 terminal, for instance, loses the sidebar and status bar but keeps the log.

When the sidebar folds, a one-row top bar reclaims HP, status-effect badges, and depth into the grid itself — show-topbar? = (not show-sidebar?) is an exact complement, so precisely one of {sidebar, top bar} exists at any width, in native TUI or WASM alike, bridge or no bridge.

Key reference: PR #131 (camera + collapse), #132 (top bar).

Touch shell, bridge & options

The browser shell gained a client-owned host with an on-screen D-pad/action UI, a named-channel bridge (ui-bridge) that mirrors title/quest/HP/depth to a browser info bar, and dark-mode handling so iOS doesn't recolor the terminal. Once the in-grid top bar existed, the HP/depth chrome duplicate was removed — title and quest are the only chrome-only data now, and there's no longer a single point of failure for HP if the bridge doesn't connect.

The info bar's OPTIONS tile is two independent reveal axes, not one ladder: tapping it (or loading with ?mode=debug) reveals player-facing chrome — font size, hold-to-repeat, build SHA. Turn/perf instrumentation needs a separate URL-only condition and can never be reached by tapping. Force-touch (for testing touch controls on a desktop) is ?mode=dev-only. Perf cells self-reveal only once real data exists, so the bar never shows dead placeholder dots.

Key reference: PR #134 (bridge), #135 (touch shell + dev-gating + OPTIONS panel).

Phone landscape

Landscape used to stack the touch shell below the terminal — the same arrangement as portrait, just rotated — which left a phone only ~130px of actual game. Short landscape viewports (max-height: 500px, so tablets and desktop force-touch testing are unaffected) now split side-by-side instead: the terminal fills the left ~70% at full height, and portrait's exact D-pad/action markup is repositioned — not rebuilt — into a ~30% right column. It rides the same touch-detection as everywhere else in the shell, so real touch users get it, not only the dev override.

Key reference: PR #137.

Resize handling & letterbox

Every resize/orientation/font-change event is debounced (trailing-edge, plus a requestAnimationFrame stage) into a single re-fit; the whole-cell pixel remainder left over after fitting is then split evenly across each axis instead of pooling wherever the browser's default box alignment happens to put it. Grid geometry itself is unchanged — only the centering offset. A resize also nudges any screen blocked on a keypress (title, death, rune-codex, help, hazard/restart/quit prompts) with a lone BEL character so it repaints at the new size; every one of those screens explicitly discards that BEL rather than treating it as a real key. Title and death screens additionally re-scatter their decorative runes on a real size change instead of freezing at whatever size they opened at, and the runes themselves stay out of a padded “moat” around the title/prompt/seed text so they can't eat characters out of it.

Key reference: PR #133 (screens reflow), #136 (fit + letterbox), #138 (nudge guard), #139 (title moat).

Verification & open items

Confirmed: the re-slice introduced no drift

Diffed integ/mobile-stack (the squashed 11-PR reference) against the current #116 tip directly. The only difference is #116's own preview-only review-doc file — zero code drift. The mechanisms described above were independently verified against the actual diffs across several earlier review passes (camera math, panel thresholds, tier-gating logic, letterbox ordering); this pass focused on what's new since — landscape-split's reachability by real touch users (confirmed: it rides the same coarsePointer || forceTouchEnabled detection as the rest of the shell, not a dev-only path) and the consolidated resize-nudge guard, which now also covers hazard/restart/quit confirmations, not only rune-codex/help/title.

Add a replay-determinism test for a camera-affecting run

The camera is render-only, keyed on depth + rect dimensions, and never touched during replay reconstruction — but nothing currently asserts that explicitly.

Add a dead-zone-edge-crossing test

Dirty-rect detection compares against the camera actually drawn last frame, not a fresh recompute — correct today, but a state-vs-recompute pattern worth locking in with a test.

Sweep sidebar/status/log thresholds across real font sizes

All three are cell-count based; the same physical window reports different cell counts at different font sizes, and a manual pass at one size won't catch a threshold bug that only shows at another.

Spot-check the landscape split on-device

The PR's own commit message flags the 70/30 ratio as “first cut, tuned on-device” — worth confirming it still reads well across a couple of real short-viewport phones before merge.

Confirm CI gates on build-info.json for the shell build specifically

Not just the composite/integration build — #135 depends on #129's output directly.
