# Dungeon terrain workbench

An exploded 3D view of stacked dungeon floors, generated by the real
`xsofy.terrain` code, with a `change generator code → regenerate → see + measure`
loop. Each floor draws as a tilted isometric layer (CSS 3D transforms + 2D canvas
— no three.js), so you can see a whole multi-floor dungeon as a tower and inspect
any level.

One page (`workbench.html`), one renderer, three modes — the page auto-detects
which, and you can force it with `?mode=`. Each carries only its own overhead:

| Mode | What drives it | Use it for |
|---|---|---|
| **static** (`?mode=static`) | render a prebuilt export — no engine | sharing a snapshot; offline / static host |
| **server** (`?mode=server`) | `terrain-server.lg`: native gen + live-redefine | fast param iteration, the dev loop |
| **wasm** (`?mode=wasm`) | the let-go image runs in-page via `LetGoHost.eval` | a shareable live playground; reset-on-reload |

Detection order, no network probe: explicit `?mode=` wins; otherwise a
`-w-host-eval` bundle (`window.LetGoHost`) → **wasm**, a page served by
`terrain-server.lg` (it injects a marker) → **server**, anything else →
**static**. Static + server expose **upload** too (drop an export into a live
session); only live modes (server/wasm) show the seed/depth controls and the
live-coding console.

## Open it

- **Static — hosted:** https://raw.githack.com/mparrett/xsofy/tool/dungeon-3d-viewer/tools/dungeon-3d/workbench.html
  (renders the default export served alongside the page).
- **Static — offline:** open `workbench.html` directly, then drop a `.json/.js`
  export on the window. (A self-contained file works too: a page carrying an
  embedded `window.FLOORS` export renders on `file://` with no fetch.)
- **Server:** `lg tools/dungeon-3d/terrain-server.lg` then open
  http://localhost:7070 — see [WORKBENCH.md](WORKBENCH.md).
- **Wasm:** build the host-eval bundle and serve it under cross-origin isolation —
  see [WORKBENCH.md](WORKBENCH.md).

The dev loop, metrics, live-coding model, and the wasm build live in
[WORKBENCH.md](WORKBENCH.md).

## Controls

- **Drag** to orbit, **scroll** to zoom, **shift/space-drag** to pan.
- **Click a floor** for a flat top-down view of just that level (floors above it
  hidden), like standing on it in-game. **Double-click** returns to the 3D stack.
- Sliders (in **view & tuning**): explode gap, tilt, spin, zoom, cell aspect,
  floor thickness, dim level.
- Per floor: **show** toggles visibility, **dim** fades it so you can see floors
  beneath. Helpers: `all`, `top` (solo the top floor), `dim ↑` (dim everything
  above the deepest shown floor).
- **load export…** (or drop a file on the window) loads a different export.

## Generating data

`export-floors.lg` dumps floors as the data the workbench reads. Run from the
xsofy root (needs the native-xxh3 `lg`):

```
lg tools/dungeon-3d/export-floors.lg            # window.FLOORS wrapper, seed 42, 5 floors
lg tools/dungeon-3d/export-floors.lg 64 8       # seed 64, 8 floors
lg tools/dungeon-3d/export-floors.lg json 64    # raw JSON, seed 64
```

Args are order-free: `json` selects raw JSON; the first bare number is the seed,
the second the floor count. The `json` form is the portable file you drop into the
upload (or serve as a static-mode default); the `window.FLOORS = {…};` form is for
embedding into a self-contained offline page (static boot honors it). Upload
accepts either. `floors-seed64.json` and `floors-seed7-d8.json` are samples;
`floors-seed7-d8.json` is the served static-mode default.

## Data format

```
{ "w":79, "h":30, "seed":42, "floors":[
    { "depth":1, "down":[x,y], "up":[x,y], "grid":[<tile codes, row-major>],
      "metrics":{…} }, … ] }
```

`grid[x + y*w]` is a tile type (0–19), mapped to glyphs/colors from a palette
inlined from `xsofy/terrain.lg` (`tile-chars` / `tile-styles`). `down`/`up` are
stair coordinates (exported; the view renders stairs straight from the grid).
`metrics` is present on live-generated floors (see WORKBENCH.md) and optional on
hand-made exports.
