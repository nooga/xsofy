# xsofy workbench

An exploded 3D view of stacked dungeon floors, generated by the real
`xsofy.terrain` code, with a `change generator code → regenerate → see + measure`
loop. Each floor draws as a tilted isometric layer (CSS 3D transforms + 2D canvas,
no three.js), so you can see a whole multi-floor dungeon as a tower and inspect
any level. It tightens the inner loop the substrate/connector proposals need to
evaluate "did this actually improve connectivity, feel, or cost."

One page (`workbench.html`), one renderer, three modes. The page auto-detects
which, and you can force it with `?mode=`. Each mode carries only its own overhead.

## Modes

All three share one renderer, one UI, and the `dungeon-build.lg` floors+metrics
builder. Only a thin Backend adapter differs, behind a `Backend` interface
(`generate`, `evalCode`) with declared `caps {live, eval, upload}`. The UI
shows/hides controls off `caps` rather than branching per mode: `live` → seed/depth
+ regenerate; `eval` → the live-coding console; `upload` → load-export/drop.

| Mode | What drives it | caps (live · eval · upload) | reset | use it for |
|---|---|---|---|---|
| **static** (`?mode=static`) | `StaticBackend` — render a prebuilt export, no engine | — · — · ✓ | n/a | sharing a snapshot; offline / static host |
| **server** (`?mode=server`) | `ServerBackend` — `workbench-server.lg`: native gen + live-redefine | ✓ · ✓ · ✓ | restart server | fast param iteration, the dev loop |
| **wasm** (`?mode=wasm`) | `WasmBackend` — the let-go image runs in-page via `LetGoHost.eval` | ✓ · ✓ · ✓ | reload page | a shareable live playground; reset-on-reload |

**Detection (no network probe):** explicit `?mode=static|server|wasm` wins;
otherwise `window.LetGoHost.eval` present (a `-w-host-eval` bundle adds it) →
**wasm**; a `window.__TERRAIN_BACKEND__='server'` marker (`workbench-server.lg`
injects it into the page it serves) → **server**; neither → **static**, the
zero-dependency default. A `-w` shell bundle exposes `window.LetGoHost` without
`.eval`, so detection gates on `.eval` specifically; an unavailable explicit
`?mode=` falls back with a dismissible notice.

## Open it

- **Static, hosted:** https://raw.githack.com/mparrett/xsofy/tool/dungeon-3d-viewer/tools/workbench/workbench.html
  (renders the default export served alongside the page).
- **Static, offline:** open `workbench.html` directly, then drop a `.json/.js`
  export on the window. A self-contained file works too: a page carrying an
  embedded `window.FLOORS` export renders on `file://` with no fetch.
- **Server:** `lg tools/workbench/workbench-server.lg`, then open
  http://localhost:7070 (see [Run](#run)).
- **Wasm:** build the host-eval bundle and serve it under cross-origin isolation
  (see [Hosted-wasm build](#hosted-wasm-build)).

## Controls

- **Drag** to orbit, **scroll** to zoom, **shift/space-drag** to pan.
- **Click a floor** for a flat top-down view of just that level (floors above it
  hidden), like standing on it in-game. **Double-click** returns to the 3D stack.
- Sliders (in **view & tuning**): explode gap, tilt, spin, zoom, cell aspect,
  floor thickness, dim level.
- Per floor: **show** toggles visibility, **dim** fades it so you can see floors
  beneath. Helpers: `all`, `top` (solo the top floor), `dim ↑` (dim everything
  above the deepest shown floor).
- **load export…** (or drop a file on the window) loads a different export.

## Why a local lg server (for dev)

`terrain.lg` is *interpreted*, so the dev loop needs no Go/wasm rebuild; only edits
to let-go's own Go code do. The native server beats the wasm path for *local* dev:
no build step (edit `.lg`, restart or live-redefine), no wasm payload, native-speed
generation, full seed control via query args. The wasm path is the *hosted/shareable*
sibling.

The server holds the let-go image; the browser page is just a view:

```
lg [-n] tools/workbench/workbench-server.lg [port]    # binds 127.0.0.1 (/eval runs code)
   ├─ GET  /floors?seed=42&depths=8  → {w,h,seed,floors:[{depth,grid,down,up,metrics}]}
   ├─ POST /eval   (body = lg code)  → evals in xsofy.terrain → {ok,value|error}
   ├─ GET  /<name>.json|.js          → a bundled export (so ?mode=static off this page renders)
   └─ GET  /                         → workbench.html (with the server marker injected)
```

## Live-coding

Two ways to redefine generator fns in the running image. Vars are late-bound, so
the next `/floors` call uses the new code with no restart and no rebuild.

- **In-browser eval console (primary).** The bottom console POSTs to `/eval`, which
  evals in `xsofy.terrain`, shows the result, and auto-regenerates. Quick-example
  buttons demonstrate it: **kill grass** (`place-grass-patch!` no-op), **kill lakes**
  (`place-lake!` no-op), **add fire** (`carve-room!` carves fire tiles, which are
  walkable and count as floor, so connectivity holds). ⌘/Ctrl+Enter evals.
- **Editor nREPL.** `lg -n` runs an nREPL on :2137 alongside the server; let-go's
  nREPL works with CIDER/Calva/Conjure (it writes `.nrepl-port`, so Conjure
  auto-connects). Eval forms from your `terrain.lg` buffer, then refetch to see them.

**The VM model (important).** The server holds the let-go image; the page is just a
view. Reloading the browser re-fetches from the same image, so redefs persist. To
reset everything, restart the server (fresh process = original source). let-go
exposes no clean in-session namespace reload (`:reload` isn't a require flag;
`load-file` doesn't resolve in the eval context), so restart is the reset today.
Wasm mode resets on reload instead, since the image lives in the page.

## Metrics (per floor)

Surfaced alongside each generated floor (tile counts already render in the viewer).
Present on live-generated floors (server/wasm); optional on hand-made exports.

- **components:** distinct reachable floor regions (flood over `floor-tiles`).
  `1` = the connectivity invariant holds; `>1` means a change broke it. The cheapest
  regression check for connectivity work.
- **dead-ends:** floor tiles with exactly one floor neighbour. A layout-density,
  "feels designed vs. maze" signal.
- **rooms:** room-center count from `generate-level`.
- **gen-ms:** per-floor `generate-level` wall time (`System/nanoTime`). Algorithm
  changes show their perf cost immediately; ties into the perf-grid harness.
- **loops:** *coarse for now.* Cyclomatic `E − V + C` over the raw tile graph, so
  open rooms (dense 4-connected meshes) inflate it to the hundreds. **Refinement:**
  compute over a room/junction graph (or count extra-connectors) for the real
  "designed loop" signal the connector proposal wants. Treat it as a placeholder.

## Generating data

`export-floors.lg` dumps floors as the data the workbench reads. Run from the
xsofy root (needs the native-xxh3 `lg`):

```
lg tools/workbench/export-floors.lg            # window.FLOORS wrapper, seed 42, 5 floors
lg tools/workbench/export-floors.lg 64 8       # seed 64, 8 floors
lg tools/workbench/export-floors.lg json 64    # raw JSON, seed 64
```

Args are order-free: `json` selects raw JSON; the first bare number is the seed,
the second the floor count. The `json` form is the portable file you drop into the
upload (or serve as a static-mode default); the `window.FLOORS = {…};` form is for
embedding into a self-contained offline page (static boot honors it). Upload
accepts either. `floors-seed64.json` and `floors-seed7-d8.json` are samples;
`floors-seed7-d8.json` is the served static-mode default.

## Data format

```
{ "w":79, "h":30, "seed":42, "floors":[
    { "depth":1, "down":[x,y], "up":[x,y], "grid":[<tile codes, row-major>],
      "metrics":{…} }, … ] }
```

`grid[x + y*w]` is a tile type (0–19), mapped to glyphs/colors from a palette
inlined from `xsofy/terrain.lg` (`tile-chars` / `tile-styles`). `down`/`up` are
stair coordinates (exported; the view renders stairs straight from the grid).
`metrics` is present on live-generated floors and optional on hand-made exports.

## Gotchas

- `/eval` runs arbitrary code, so it's hardened against cross-origin abuse: the
  server binds `127.0.0.1` only, sends no `Access-Control-Allow-Origin`, and rejects
  any request whose `Origin` isn't the server's own (a page on another origin can't
  POST code to it). Requests with no `Origin` (curl, local tooling) still pass.
- `/<name>.json|.js` is served by basename only (no `/` or `..`), so the export route
  can't reach `.lg` source or anything outside `tools/workbench/`.
- `[http :refer :all]` refers `http/get`, which shadows core `get`; use `[http :as
  http]`. (Bit us with a flaky `/floors` 500.)
- `loops` is coarse (see Metrics).

## Run

```
# from the xsofy root (or the worktree at the workbench branch)
lg tools/workbench/workbench-server.lg            # serve on :7070, open http://localhost:7070
lg tools/workbench/workbench-server.lg 8080       # custom port
lg -n tools/workbench/workbench-server.lg         # + nREPL :2137 (editor live-coding)
```

Open the page, set/step seed & depths (auto-regenerates), and expand the
**live-coding** console to eval into the running image. Tracks let-go `main`
(native xxh3 + `*command-line-args*` + `http`/`io`/`json`/`System`), same as the
rest of the tool.

### Hosted-wasm build

Wasm mode runs the let-go image *in the page*, so reload = fresh VM (reset for free)
and the workbench can host statically (no local server) for shareable demos.
`WasmBackend` drives it through the public `LetGoHost.eval(code)` API; `generate`
and `eval-str` return their payloads out of band via `js/emit` (the `xsofy/floors`
and `xsofy/eval-result` events), since `LetGoHost.eval` only hands back a stringified
value. Validated end-to-end in a browser: floors render in-page and the console
redefines into the live `xsofy.terrain` image, no server.

```
lg -w dist -w-shell none -w-host-eval -w-wasm external tools/workbench/wasm-entry.lg
# inject workbench.html into dist/index.html, then serve dist/ with COOP/COEP headers
# and open  …/index.html?mode=wasm
```

- **let-go requirement:** wasm mode needs a let-go built with **`-w-host-eval`**
  (shipped in the v1.11.0 release). Server mode needs only let-go `main`.
- **COI:** a generator-only wasm never calls `read-key`, so it can run on the main
  thread without cross-origin isolation (the static-hosting path). `LetGoHost.eval`
  is dual-mode (main-thread direct or worker relay under COI), so both work;
  validated so far under COI (worker mode).
