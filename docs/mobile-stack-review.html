Executive summary

This stack makes xsofy playable on small and touch viewports without changing the deterministic simulation. A dead-zone camera follows the player once the world no longer fits the visible map, and per-panel width/height thresholds drop the sidebar, log, and status bar independently rather than as one unit (#112). A one-row in-grid top bar reclaims HP/status/depth into the terminal grid itself whenever the sidebar folds, so that data has a native-and-WASM home even if the browser bridge never connects (#119). A native engine benchmark ships alongside, unrelated to layout (#111). The shell gained a touch UI, a named-channel HUD bridge, a build-info manifest, and a title-screen input guard (#113); a tiered ?mode= ladder that separates player options from debug/dev tooling (#120); and the removal of the now-redundant HP/depth readout from the browser chrome, safe only once the top bar exists (#121, depends on #119). Resize handling was rebuilt in two steps: #113's shell rewrite dropped the debounce and the orientationchange/visualViewport listeners that existed pre-stack, and #114 restores them — not merely extends them — while also adding the missing visualViewport scroll listener and a requestAnimationFrame stage. #115 then adds pixel-level letterbox centering on top. Two later additions round out the round: the animated title and death screens now re-read the terminal size and repaint on a resize/font-change instead of freezing at the size they opened at (#122), and the browser info bar is reworked so a single OPTIONS tile sits alone at the top-right while the perf stats and the player-option chips (font, hold-to-repeat, force-touch) share one row beneath it, and the turn/perf/dimension diagnostics self-hide until they carry a real value (#123). A separate title-screen fix keeps the decorative runes off the title / prompt / seed text they were blanking as they pulsed (#124). PR #116 is the integration build that bundles all of #111–#115 and #119–#123, with #124 folded into the same preview.

Most important finding from reading the source directly

The pre-stack map area was 80×26 cells even at a common 30-row terminal height — well under the design cap of 100×36. xsofy's dungeon floors are 79×30 cells (per the camera module's own comment), so the bottom 4 rows of every floor were invisible and unreachable at any terminal shorter than 34 rows — i.e. nearly always in practice, not only on unusually small terminals. This is the concrete bug #112 fixes; it is a correctness fix, not only a responsiveness improvement. (Unaffected by this revision's new PRs — #119/#120/#121 touch HP/status display, not the world-render clip.)

Before the stack

Verified directly from the PR #112 base commit (dc567de), not inferred from behavior. Three files matter: xsofy/ui.lg (layout), xsofy/render.lg (map/camera), and tools/xsofy-shell.html (browser host).

  • No camera. render-map iterated world coordinates 0..min(width, rect.w) directly onto screen coordinates — zero offset. Anything past the map rect's edge, including the player, was simply never drawn.
  • Fixed-size chrome, always shown. Sidebar (20 cols, left-hand side), log (3 rows), and status (1 row) were unconditional — no threshold, no collapse. On a small terminal the sidebar could consume most of the available width.
  • The game rect already centered itself. layout computed ox/oy via integer-divide centering of the fixed 100×36 rect inside the terminal. This was not pinned top-left, and #112 did not need to fix it.
  • Resize was already debounced. The 88-line pre-stack shell already wired resize, orientationchange, and visualViewport resize through a 100ms setTimeout debounce directly into fit.fit(). This did not need to be invented by #114.
  • No letterbox. The old refit() never touched padding, so the whole-cell pixel remainder from fit.fit() sat wherever the browser's default box alignment put it.
  • No touch input, no HUD bridge. The shell had zero touch-related markup or JS; input was keyboard-only.

The actual pre-stack layout at three terminal sizes, drawn to the real fixed dimensions (sidebar 20 cols, log+status 4 rows, both always rendered). At 30×30 the sidebar alone consumes two-thirds of the terminal width.

Trigger conditions

The five mechanisms are independent axes — none of them gate each other. A touch laptop in a maximized window gets a touch shell and the full sidebar. A keyboard-only SSH session into a tiny pane gets chrome collapse, a top bar, and camera panning with no touch shell at all.

Sidebar, log, and status collapse independently — not as one unit

This matters for review: a 40×24 terminal hides the sidebar and the status bar (both fail their width checks) but keeps the log, because 24 rows clears the log's 14-row threshold. Testing only a single “narrow” size will not exercise all three thresholds independently — use sizes that straddle each one on its own.

HP always has an in-grid home — verified, not assumed

show-topbar? = (not show-sidebar?) is an exact boolean complement, not an approximation: at every width, precisely one of {sidebar, top bar} is present. There is no gap size where neither exists. Traced the row math through rect-split-v directly (it offsets the y-coordinate of the remainder, so the top bar and map area never overlap) rather than trusting the commit message.

Sidebar and top bar are exact complements of each other, and the top bar's row is carved out before the sidebar/map/log/status split, not overlaid on top of it — so the map area's height genuinely shrinks by one row rather than the top bar drawing over the first map row.

Design note — giving HP a home (#119)

Why a full PR for one row: below 80 columns the sidebar folds, and pre-stack that left HP surviving only in the browser chrome — invisible to SSH and native TUI, and gone entirely if the HUD bridge never connected. The top bar keeps the survival-critical read in the terminal grid itself, at every width and on every host. It carries only what a player consults mid-turn; the sidebar's reference data stays in the wide layout.

The full sidebar shows eleven things; only the top three are read between keystrokes. The bar keeps those three and lets the wide layout keep the rest.

  • Belongs in the bar. HP — the number you die by, anchoring the row as a draining strip. Status effects — poison/burning tick damage, frozen skips a turn; the color is the signal. Depth — cheap orientation, two cells.
  • Stays in the wide sidebar. Str/Def, weapon & armor, gold, seed, and the visible-creature threat list — the last already legible on the map as glyphs, so it doesn't need a second home.

Two treatments. The recommended form (variant B) gives the row's weight to an HP strip that drains left-to-right, with status badges clustered on the right, each in its own color — a green pip reads as poison at a glance instead of hiding in monochrome text. The ultra-narrow floor (variant A, under ~24 cols) drops the badge cluster to a single ! that flags “something's ticking” without spending a cell per effect — the smartwatch case, not the default.

What sheds, and in what order. As width shrinks the bar sheds from the right — threat count first, then status badges, then depth — so the HP strip is the last thing standing. Same discipline as the panel collapse, applied within a single row. Top bar and sidebar stay exact complements: never both at once, so there's no double vision.

A fuller exploration, with live terminal mockups at each width, was produced during design; it isn't committed to the tree.

How the pieces compose at runtime

Per-PR review can't show this: on every resize, sizing flows through #114 → #115 → #112(+#119) in sequence. #113 is orthogonal to camera and layout — it never touches those — but it is not orthogonal to resize handling itself: #113's shell rewrite is where refit()'s grid-change gate and BEL redraw-nudge originate, and where the pre-stack debounce and orientationchange/visualViewport listeners got dropped. #114 restores the latter and adds the debounce/rAF wrapper and the missing scroll listener around the former, which it reuses unchanged.

The 40×24 case is worth checking specifically: chrome collapse fires there, but if the world still fits at 40 cells wide the dead-zone camera should not be panning — the width threshold (#112 layout) and the world-vs-viewport check (#112 camera) are two separate computations that happen to usually agree. A regression that only affects one of them would be easy to miss without checking both explicitly.

Cross-cutting note: chrome collapse and the camera are both driven by cell counts; the touch shell is driven by input capability; letterbox is driven by leftover pixels after flooring to cells. The same physical window can land in different states at different font sizes, since font size changes the cell count without changing the pixel size.

Follow-up tasks before wider review

Concrete items surfaced while reading the actual diffs and source, not general polish requests — carried forward from rev. 1, re-verified against origin, and extended with an independent review of #119/#120/#121. Items 3 and 11 are no longer patches attached to this doc — they're committed upstream and verified byte-identical to what rev. 1 proposed. Item 4 is resolved by #119+#121, independently re-checked rather than taken on trust. Items 14–16 were new in rev. 2; items 18–23 are new in rev. 3 — the #123/#124 pass plus one gap this review caught in its own rev. 2 (#120, item 18).

1. Add a replay-determinism test that covers a camera-affecting run (#112)

last-camera is global mutable render state, keyed on :depth + rect dimensions and never touched during replay reconstruction — render-only by construction. But no test currently asserts “replay renders identically whether or not a camera shift occurred.” Cheap to add; closes a real coverage gap rather than a suspected bug.

2. Add a dead-zone-edge-crossing test for render-dirty's camera-change detection (#112)

render-dirty compares against the camera that was actually drawn last frame (@last-camera), not a fresh recompute from prev-world — correct today, but a state-vs-recompute pattern that could regress silently if a future refactor swaps in a recomputed comparison. Add a test that crosses the dead-zone boundary and asserts the full-redraw path fires, not the FOV-delta path.

3. RESOLVED IN STACK — rune-codex/help BEL-dismiss guard, committed on #113 (a9b1cd7)

Rev. 1 flagged that :rune-codex and :help dismissed on a bare (term/read-key) with no BEL filtering, unlike the guarded title screen, and attached a patch. That patch is now committed directly on #113's branch as a9b1cd7 — diffed against the attached patch and confirmed byte-identical (same read-dismiss-key! helper, same two call sites). No other run-loop screen was found unguarded; the standalone patch file can be discarded.

4. RESOLVED — HUD-bridge-only HP risk closed by #119 + #121, independently re-verified (#113/#119/#121)

Rev. 1 flagged the browser HUD bridge as the only HP source on mobile, with no fallback if it failed to connect. #119 puts HP/status/depth in the terminal grid itself (native and WASM alike) whenever the sidebar folds, and #121 removes the now-redundant chrome copy. Re-derived independently rather than trusting the PR description: show-topbar? = (not show-sidebar?) is an exact complement of the sidebar condition, so at every width exactly one of {sidebar, top bar} is present — there is no width where HP has no grid home, bridge or no bridge. Title and quest remain chrome-only (no grid space for them), which is an accepted, unchanged tradeoff, not a new gap.

5. Confirm the world-window fix (2c29609) actually closes the edge-drop gap (#112)

It landed after the other three #112 commits and after the PR's own “Why” section was written — it reads like a fix for exactly the edge-drop bug the PR describes. Confirm it fully closes the gap rather than narrowing it, since it's the newest and least-reviewed piece of #112.

6. Verify the letterbox clear-then-measure ordering can't regress (#115)

refit() clears padding, measures, then reapplies. If a future change reorders that, the fit math measures the padded box instead of the full one and centering drifts on every resize. Worth a regression test that asserts padding is zero at the moment fit.fit() is called.

7. Check that refit()'s grid-change gate short-circuits before layout math, not after (#113, reused by #114)

The gate itself is #113's, not #114's — #114 only wraps the existing refit() in its debounce. It's what makes constant iOS visualViewport scroll events (fired as the URL bar hides) cheap no-ops. If the gate only skips the setSize/BEL call and still recomputes letterbox padding every time, the debounce controls call frequency but not actual cost.

8. Exercise camera vertical panning at the common 30-row size, not only small terminals (#112)

Since the map area (80×26) is smaller than the world (79×30) even at this common size, the dead-zone camera is active there too. QA scripts that only test “small vs. large” terminals will miss this; it needs its own pass at the default size.

9. Test the desktop narrow half-screen path explicitly, not just TUI sizes (#112 code path)

A tiled/half-screen browser window and a 30×30 xterm exercise the identical chrome-collapse and camera code path — now also the identical top-bar-activation path (#119). A fix validated only against one input could still leave the other broken.

10. Confirm the individual #113 pipeline gates on build-info.json, not only #116's

e07db64 is a #113 commit — #113 alone produces build-info.json, it isn't specific to the #116 composite build. Relevant to the smoke-gate 404 already being tracked separately (R2 fix) — cross-referencing here so the CI-gate check isn't scoped only to the integration build.

11. RESOLVED IN STACK — stale threshold comments in layout, committed on #112 (cfe6f63)

Rev. 1 flagged the log threshold's comment (said “<16 rows”; the actual check is gh ≥ 14) and the sidebar comment's ambiguous “~60 cols.” Committed directly on #112's branch as cfe6f63 — diffed against the attached patch and confirmed byte-identical. The standalone patch file can be discarded.

12. No action needed — render-sidebar/-log/-status use three different nil-handling strategies, confirmed safe

render-sidebar has an explicit (when r ...) guard; render-status has none but every call goes through ui/write-line, which guards internally; render-log has no guard at all. Verified safe as-is against the pinned lg v1.11.1 build. Flagging only so a future “consistency cleanup” doesn't change behavior while making the code look more uniform.

13. Verify sidebar/log/status thresholds against real font-size changes (#112)

All three collapse thresholds are cell-count based, but the same physical window reports different cell counts at different font sizes. A manual QA pass at one font size won't catch a threshold bug that only appears at another; the font-cycle debug control is the fastest way to sweep this.

14. NEW — update the #114 PR description to say “restores,” not “extends” (#114)

Confirmed by direct diff: at #113's tip, refit() called fit.fit() synchronously with a single bare resize listener — no debounce, no orientationchange, no visualViewport. #114 doesn't extend existing coverage; it puts back what #113's shell rewrite (copying in code from a separate “dev shell”) silently dropped relative to the pre-stack baseline, then adds the genuinely new scroll listener and rAF stage on top. Documentation-only ask — no functional issue — but future readers of the history should see the regression-then-restoration accurately.

15. Minor — .mode-play is now a vestigial CSS class in the shell (#120)

<div id="xsofy-shell" class="mode-play"> is still static markup, but no CSS selector in the tier-gating rewrite references .mode-play anymore — visibility is entirely driven by :not(.mode-debug) / :not(.mode-dev). Harmless today; flagging so nobody assumes removing it changes behavior, and so nobody assumes it's still load-bearing.

16. Forward-looking — top-bar status badges truncate to 3 chars with no collision guard (#119)

topbar-badge-text takes the first 3 characters of each status effect's name. Checked all 8 defined effects in status.lg — no two share a 3-letter prefix today, so this isn't a live bug. It has no uniqueness check, though; a future status effect sharing a prefix with an existing one (e.g. a second “con-” effect) would render two visually identical badges. Worth a comment, or a cheap test that asserts prefix-uniqueness across status-defs, before the effect roster grows much further.

17. No action needed — real touch-shell visibility is unaffected by ?mode=dev-gating (#120)

Worth checking directly since it would have been a severe regression: the actual on-screen touch controls are shown by an @media (orientation: portrait) rule and a separate body.force-touch class, neither of which depends on .mode-dev. Only the manual override button for force-touch testing is gated behind the dev tier — real touch/portrait users get the touch shell exactly as before, with no ?mode= param needed.

18. RESOLVED, but missed in rev. 2 — ?mode=dev could show less chrome than ?mode=debug after a tap (#120)

Own up to this one: rev. 2 verified that visiting ?mode=dev correctly set both mode-dev and mode-debug at page load, and stopped there. It didn't check what happens after tapping the OPTIONS tile post-load — which cycles currentShellState through ["play","debug"] independently of the URL-derived devMode constant, so a tap could drop to “play” and remove mode-debug while mode-dev stayed on — dev tier showing less than debug tier.
